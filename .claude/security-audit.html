Tourly — Security Audit Report

June 22, 2026 · Expo 54 + Supabase + InstantDB · React Native mobile app

1. Security Posture Rating

🟠 NEEDS WORK

Tourly has a solid foundation — RLS is enabled with WITH CHECK on core tables, secrets stay server-side in edge functions, and auth uses Supabase's built-in password + OAuth flows. However, two edge functions (parse-excel and map-import-columns) have no authentication at all, the shared-data feature bypasses RLS by querying another user's data without verifying the share relationship, offline mutations skip auth checks, and the useRevokeShare mutation lacks ownership validation. These are exploitable gaps that need fixing before production.

2. Critical & High Findings

HIGH #1 — Unauthenticated Edge Functions

Category

Missing Authentication

CWE

CWE-306 (Missing Authentication for Critical Function)

Location

supabase/functions/parse-excel/index.ts:4
supabase/functions/map-import-columns/index.ts:4

What's wrong

Both parse-excel and map-import-columns accept requests from anyone — no Authorization header check, no JWT validation. Any user who knows the function URL can call them.

Why it matters

map-import-columns calls the Anthropic API with user-provided data. An attacker could spam it to exhaust your Anthropic API credits. parse-excel could be used to process arbitrarily large files, consuming Supabase function compute time.

The vulnerable code (parse-excel)

serve(async (req) => {
  try {
    const { base64, fileName } = await req.json();
    // No auth check — jumps straight to parsing
    ...

The fix

serve(async (req) => {
  try {
    const authHeader = req.headers.get('Authorization');
    if (!authHeader) {
      return new Response(JSON.stringify({ error: 'Unauthorized' }),
        { status: 401, headers: { 'Content-Type': 'application/json' } });
    }
    const supabase = createClient(
      Deno.env.get('SUPABASE_URL')!,
      Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
    );
    const jwt = authHeader.replace('Bearer ', '');
    const { data: { user }, error } = await supabase.auth.getUser(jwt);
    if (error || !user) {
      return new Response(JSON.stringify({ error: 'Unauthorized' }),
        { status: 401, headers: { 'Content-Type': 'application/json' } });
    }

    const { base64, fileName } = await req.json();

HIGH #2 — Shared Data Bypasses RLS / No Access Verification

Category

Broken Access Control

CWE

CWE-862 (Missing Authorization)

Location

hooks/useSharedAccess.ts:104-125

What's wrong

useSharedData(ownerId) directly queries another user's tournaments, expenses, and profile by their user_id. It does not verify that the current user has an accepted share relationship. Meanwhile, the core RLS policies (auth.uid() = user_id) should block this entirely — meaning either (a) this feature is broken and returns empty data, or (b) there's an additional RLS policy granting broader read access that we can't verify from client code alone.

Why it matters

If RLS allows this query, any authenticated user can read any other user's complete financial data by guessing or knowing their UUID. If RLS blocks it, the feature is silently broken — a different kind of problem.

The vulnerable code

export function useSharedData(ownerId: string | null) {
  return useQuery({
    queryFn: async () => {
      // No check that current user has accepted share access
      const [tournamentsRes, expensesRes, profileRes] = await Promise.all([
        supabase.from('tournaments').select('*').eq('user_id', ownerId),
        supabase.from('expenses').select('*').eq('user_id', ownerId),
        supabase.from('profiles').select('...').eq('id', ownerId).maybeSingle(),
      ]);
      ...

The fix

export function useSharedData(ownerId: string | null) {
  return useQuery({
    queryFn: async () => {
      const { data: { user } } = await supabase.auth.getUser();
      if (!user || !ownerId) return null;

      // Verify share relationship exists and is accepted
      const { data: share } = await supabase
        .from('shared_access')
        .select('id')
        .eq('owner_id', ownerId)
        .eq('shared_with_id', user.id)
        .eq('status', 'accepted')
        .maybeSingle();
      if (!share) throw new Error('Access denied');

      // Use the shared_tournaments / shared_expenses views instead
      const [tournamentsRes, expensesRes, profileRes] = await Promise.all([
        supabase.from('shared_tournaments').select('*')
          .eq('user_id', ownerId).eq('shared_with_id', user.id),
        supabase.from('shared_expenses').select('*')
          .eq('user_id', ownerId).eq('shared_with_id', user.id),
        supabase.from('profiles').select('full_name, ranking, nationality')
          .eq('id', ownerId).maybeSingle(),
      ]);
      ...

HIGH #3 — Offline Mutations Skip Authentication

Category

Missing Authentication

CWE

CWE-287 (Improper Authentication)

Location

lib/api.ts:20-24, :32-35, :49-51, :66-68, :78-80

What's wrong

Every API function checks connectivity first. When offline, it enqueues the mutation without checking auth — the auth check only runs in the online branch. When the queue processes later (offline-queue.ts:45), it does check auth, but the enqueued mutation's data field never includes a user_id — the queue's executeMutation adds user_id from the current session. If the user signs out and another user signs in on the same device, the queued mutations from User A would execute under User B's identity.

Why it matters

Data from one user's session could be written to another user's account on shared/borrowed devices.

The vulnerable code

export async function apiAddTournament(data: Record<string, any>) {
  if (await isOffline()) {
    await enqueue({ table: 'tournaments', action: 'insert', data });
    return data; // No auth check — queued blindly
  }
  // Auth check only happens in the online path
  const { data: { user } } = await supabase.auth.getUser();
  ...

The fix

export async function apiAddTournament(data: Record<string, any>) {
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) throw new Error('Not authenticated');

  if (await isOffline()) {
    await enqueue({
      table: 'tournaments', action: 'insert', data,
      userId: user.id,  // Pin the user identity at enqueue time
    });
    return data;
  }
  ...

~20 minutes (update all 5 API functions + queue schema + executeMutation to compare userId)

HIGH #4 — useRevokeShare Lacks Ownership Check

Category

Broken Access Control

CWE

CWE-862 (Missing Authorization)

Location

hooks/useSharedAccess.ts:90-101

What's wrong

useRevokeShare deletes a shared_access row by id alone — it doesn't filter by owner_id = auth.uid(). While the RLS DELETE policy (auth.uid() = owner_id) would prevent cross-user deletes at the database level, the client code doesn't signal intent correctly. If a future RLS change loosens the policy, this becomes exploitable.

Why it matters

Defense-in-depth: the client should always scope mutations to the current user. Currently the only protection is the RLS policy.

The fix

mutationFn: async (shareId: string) => {
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) throw new Error('Not authenticated');
  const { error } = await supabase
    .from('shared_access')
    .delete()
    .eq('id', shareId)
    .eq('owner_id', user.id);  // Scope to current user
  if (error) throw error;
},

MEDIUM #5 — shared_access UPDATE Policy Too Broad

Category

Broken Access Control

CWE

CWE-863 (Incorrect Authorization)

Location

supabase/add_shared_access.sql:29-31

What's wrong

The UPDATE policy allows both shared_with_id and owner_id to update any column. A shared user (invitee) could change their own role from viewer to editor, or change the status or owner_id.

The vulnerable code

CREATE POLICY "Shared users can accept invites"
  ON shared_access FOR UPDATE
  USING (auth.uid() = shared_with_id OR auth.uid() = owner_id);
  -- No WITH CHECK, no column restriction

The fix

-- Invitees can only accept (update status + set their id)
CREATE POLICY "Shared users can accept invites"
  ON shared_access FOR UPDATE
  USING (auth.uid() = shared_with_id)
  WITH CHECK (
    status = 'accepted'
    AND shared_with_id = auth.uid()
    AND role = 'viewer'  -- prevent privilege escalation
  );

-- Owners can update their own shares
CREATE POLICY "Owners can update their shares"
  ON shared_access FOR UPDATE
  USING (auth.uid() = owner_id)
  WITH CHECK (auth.uid() = owner_id);

~10 minutes

MEDIUM #6 — No Rate Limiting on parse-excel and map-import-columns

Category

Missing Rate Limiting

CWE

CWE-770 (Allocation of Resources Without Limits)

Location

supabase/functions/parse-excel/index.ts
supabase/functions/map-import-columns/index.ts

What's wrong

map-import-columns calls the Anthropic API on every request with no rate limit. An attacker (or even an overeager user) could call it repeatedly, racking up API charges. parse-excel has no file size limit — a 50MB base64 blob would consume significant function memory.

The fix

After adding auth (Finding #1), add a simple database-backed rate limit like the other edge functions use. For parse-excel, add if (base64.length > 10_000_000) return 413; (roughly 7.5MB file).

~15 minutes

MEDIUM #7 — recommend-tournaments Has No Rate Limiting

Category

Missing Rate Limiting

CWE

CWE-770 (Allocation of Resources Without Limits)

Location

supabase/functions/recommend-tournaments/index.ts

What's wrong

This function has proper auth but no rate limiting (unlike generate-insight/coaching/reflection). It calls the Anthropic API with max_tokens: 1024 per request. An authenticated user could spam it.

The fix

Add the same rate-limit pattern used in generate-insight — query the last recommendation timestamp, enforce a cooldown (e.g., 60 minutes).

LOW #10 — Hardcoded InstantDB App ID Fallback

Category

Hardcoded Configuration

CWE

CWE-798 (Use of Hard-coded Credentials)

Location

db.ts:28

What's wrong

The InstantDB App ID has a hardcoded fallback: 'f819fcd1-f0da-4658-ac5c-a190539808f6'. While an app ID is not a secret (it's a public identifier), hardcoding it means the app silently connects to this database even if the env var is misconfigured. This is especially risky if InstantDB has no server-side auth rules, as the app ID alone grants read/write access.

The fix

const APP_ID = process.env.EXPO_PUBLIC_INSTANTDB_APP_ID;
if (!APP_ID) throw new Error('EXPO_PUBLIC_INSTANTDB_APP_ID is required');

5. What's Already Done Right

  • RLS on all tables — Every table (tournaments, expenses, profiles, coaching_messages, reflections, insights, match_results, training_blocks, shared_access) has RLS enabled with proper auth.uid() policies and WITH CHECK clauses
  • Secrets stay server-side — ANTHROPIC_API_KEY and SUPABASE_SERVICE_ROLE_KEY are only accessed via Deno.env.get() in edge functions; never exposed to the client
  • .env properly gitignored — .env and .env.* covered; no .env files in git history
  • EXPO_PUBLIC_ prefix used correctly — Only truly public values (Supabase URL, anon key, InstantDB app ID, demo mode flag) use the public prefix
  • Auth uses getUser() not getSession() — All API functions and edge functions validate via supabase.auth.getUser() (server-validated JWT), not the client-only getSession()
  • Rate limiting on AI functions — generate-insight (60 min), generate-coaching (6 hours), generate-reflection (60 min) all have database-backed cooldowns with 429 responses
  • Input allowlisting — generate-insight validates trigger and forced_type against whitelists; generate-reflection validates period_type
  • user_id from session, not request body — All insert operations derive user_id from the authenticated session, never from client-supplied data
  • Defense-in-depth filtering — Edge functions add .eq('user_id', user.id) to queries even though RLS already enforces it
  • No XSS vectors — No dangerouslySetInnerHTML or equivalent found anywhere in the codebase
  • Supabase query builder prevents SQL injection — No raw SQL or string concatenation in client code; parameterized queries throughout
  • Generic error messages on auth functions — generate-insight, generate-coaching, generate-reflection all return "Internal server error" (not err.message)
